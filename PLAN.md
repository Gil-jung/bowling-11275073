# PLAN — Cycle 4: 10프레임 스트라이크 보너스 롤

## 0. 이전 사이클

- Cycle 1(거터 게임): `score() = sum(rolls)` — 완료
- Cycle 2(스페어 보너스): 프레임을 2롤씩 순회, 스페어 시 `10 + 다음 1롤` — 완료
- Cycle 3(스트라이크 보너스): 스트라이크 시 `10 + 다음 2롤`, 인덱스 1 전진 — 완료

현재 `score()`는 프레임 경계를 별도로 구분하지 않고, "현재 프레임의 첫 롤" 위치에서
다음 1~2개의 롤을 그대로 읽어와 보너스로 더하는 **평평한(flat) 인덱스 기반** 순회다.

## 1. 목표 (RED 1단계)

**단 하나의 동작**: 10프레임에서 스트라이크가 나오면, 같은 프레임 안에서
2번의 보너스 롤이 추가로 던져지고, 그 두 롤이 그대로 보너스로 합산된다
(그 이후 더 이상의 프레임/롤은 없다).

## 2. 작성할 실패 테스트 — 그리고 중요한 사전 고지

- 파일: `test_bowling.py`
- 테스트: `test_tenth_frame_strike_gets_two_bonus_rolls`

```python
def test_tenth_frame_strike_gets_two_bonus_rolls():
    game = Game()
    for _ in range(9):
        game.roll(0)
        game.roll(0)   # 1~9프레임: 전부 거터
    game.roll(10)      # 10프레임: 스트라이크
    game.roll(3)       # 보너스 롤 1
    game.roll(4)       # 보너스 롤 2

    # 1~9프레임: 0
    # 10프레임: 10 + 3 + 4 = 17
    assert game.score() == 17
```

**⚠️ 사전 고지 (투명하게 밝힘):** 현재 구현은 프레임 번호를 특별 취급하지 않고
"다음 1~2롤"을 그냥 리스트 인덱스로 읽기 때문에, 시뮬레이션해본 결과 이 테스트는
**코드 변경 없이 이미 통과할 가능성이 높다.** (10프레임의 보너스 롤이든 다음 프레임의
정규 롤이든, 현재 알고리즘 입장에서는 둘 다 "다음 인덱스의 값"으로 동일하게 처리되기
때문.)

만약 실행 결과 실제로 이미 GREEN이라면:
- 이는 Cycle 2·3에서 만든 범용 로직이 10프레임 경계까지 우연이 아니라 구조적으로
  이미 올바르게 처리하고 있다는 뜻이다.
- 이 경우 GREEN 단계에서 프로덕션 코드는 변경하지 않고, 이 테스트를 **회귀
  방지용 확인 테스트**로만 커밋한다 (Iron Law 위반 아님 — 실패하는 테스트를
  먼저 작성했고, 그 결과 이미 올바르다는 것을 "발견"한 것).
- 만약 예상과 달리 실패한다면(예: 10프레임을 감지 못해 11번째 "프레임"을 시도하며
  `range(10)` 밖 인덱스 접근 등), 그 실패를 기준으로 정상적인 GREEN 구현을 진행한다.

## 3. 구현 방향 (필요할 경우에만, GREEN에서 진행)

테스트가 실제로 실패할 경우에만: 10프레임 도달 여부를 프레임 카운트가 아니라
"남은 롤 수"로 판단해, 스트라이크/스페어 보너스 롤을 정규 프레임으로 오인해
추가 순회하지 않도록 종료 조건을 보정한다.
